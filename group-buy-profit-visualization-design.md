# 团购盈利能力可视化开发设计实施文档

## 1. 文档目标

本文档用于指导 `fresh-shop` 在当前已有数据模型和统计口径基础上，优化“团购盈利能力”的数据可视化与交互展示方案。

本次方案严格遵循以下边界：

- 不新增数据库字段
- 不新增后端统计接口字段
- 不调整现有利润、退款、订单量统计口径
- 不扩展到当前需求之外的经营分析能力

本文档聚焦“如何基于当前已有字段，把盈利能力展示得更清楚、更可比较、更可分析”。

## 2. 现状结论

### 2.1 当前已具备的数据基础

当前项目已经具备支撑“团购盈利能力展示”的核心数据基础：

- 团购规格 `units` 中已有：

  - `totalRevenue`

  - `totalProfitMargin`
  - `totalOrderCount`
  - `uniqueCustomerCount`
  - `totalRefundAmount`
- 团购概况详情已有：
  - `totalRevenue`

### 2.2 当前盈利展示的主要问题

虽然数据基础足够，但当前展示方式仍偏“静态结果汇总”，主要存在以下问题：

1. 规格层只展示价格信息，没有直观呈现“定价能力”

- 当前团购详情页中的规格区只展示：
  - 规格名
  - 售价
  - 成本价
- 用户需要自行心算单份毛利和毛利率，理解成本高

2. 团购概况列表适合查找，不适合发现盈利异常

- 当前列表对单个团购的销售额、利润、利润率、订单量、退款金额展示完整
- 但缺少二维比较视角，用户不容易快速识别：
  - 高销售低利润率
  - 低销售高利润率
  - 退款拖累严重
  - 规模大但盈利差

3. 团购详情弹窗以统计卡片为主，缺少“盈利拆解”表达

- 当前详情页能看到总销售额、总利润、利润率、订单量、发起次数、退款金额
- 但用户仍然看不清“利润为什么高/低”
- 缺少把退款影响、利润效率、期次波动串起来的表达

4. 历史数据以表格为主，不利于识别盈利波动趋势

- 当前 `groupBuyLaunchHistory` 表格适合精读
- 但不适合快速判断哪几期表现异常、退款冲击是否集中、利润率是否稳定

5. 客户与区域数据存在，但当前层级略靠前

- 客户忠诚度和区域分布是解释性信息
- 它们适合回答“为什么这款团购表现这样”
- 但不应抢在“盈利本身是否健康”之前占据主舞台

## 3. 设计原则

本次改造遵循以下原则：

### 3.1 先回答盈利，再回答原因

用户关注盈利能力时，优先想知道的是：

1. 赚不赚钱
2. 是靠规模赚钱还是靠利润率赚钱
3. 退款有没有明显侵蚀利润
4. 哪几期表现好，哪几期表现差
5. 当前规格定价有没有明显问题

因此页面顺序必须先展示“盈利结果”，再展示“客户/区域解释因素”。

### 3.2 不做伪精细化

当前字段不能支持严格的规格级经营归因，因此不能硬做：

- 规格级真实退款归因
- 规格级真实利润贡献图
- 团购固定成本后的净利润图

在当前字段下，规格层更适合展示“定价能力”，而不是“经营归因结果”。

### 3.3 图表优先承担比较，表格承担明细

- 图表负责发现模式、异常、结构差异
- 表格负责看精确数字、排序、跳转明细

### 3.4 所有展示遵循现有统计口径

本次所有金额和利润展示均沿用现有统一口径：

- 退款订单销售额为 `0`
- 非退款订单销售额为 `原始金额 - partialRefundAmount`
- 非退款订单利润为 `原始利润 - partialRefundAmount`
- 退款订单利润为 `- 原始成本`
- 订单量仅统计 `PAID / COMPLETED`

## 4. 核心设计方案

本次方案分为四个展示层：

1. 规格定价能力层
2. 团购概况比较层
3. 团购详情盈利分析层
4. 总览趋势效率层

## 5. 规格定价能力层

### 5.1 适用页面

- `frontend/src/pages/GroupBuy/Detail.tsx`

### 5.2 当前问题

当前“所有规格”区域是静态卡片，适合查看原始字段，但不适合比较。

### 5.3 优化目标

让用户在不新增任何后端字段的前提下，直接看出：

- 哪个规格单份更赚钱
- 哪个规格毛利率更高
- 哪个规格看起来卖得贵，但其实毛利率不高

### 5.4 具体方案

将当前规格卡片区改为“规格定价能力表 + 视觉对比”双层结构。

#### 方案 A：规格定价能力表

表格字段：

- 规格
- 售价
- 成本价
- 单份毛利 = `price - costPrice`
- 毛利率 = `(price - costPrice) / price`

展示要求：

- 默认按“单份毛利”降序排序
- 支持切换按“毛利率”排序
- 利润为正使用绿色系
- 单份毛利接近 0 的规格使用橙色提醒
- `price < costPrice` 的规格延续当前已有风险提醒逻辑

#### 方案 B：规格毛利横向对比条

在表格上方或右侧增加一个轻量横向条形对比：

- Y 轴：规格名
- X 轴：单份毛利
- 条颜色：
  - 正毛利：绿色
  - 低毛利：橙色
  - 负毛利：红色

### 5.5 为什么这样做

当前没有规格级经营结果字段，因此不能讲“这个规格实际贡献了多少利润”。

但当前字段足以讲“这个规格的定价本身是否合理”。这是当前数据边界内最可信、最直接、最可执行的展示方式。

## 6. 团购概况比较层

### 6.1 适用页面

- `frontend/src/pages/Analysis/components/GroupBuyOverview/index.tsx`

适用于：

- 合并同名模式
- 单期模式

### 6.2 当前问题

当前页面以列表为主，适合搜索和精读，但不适合宏观发现问题。

### 6.3 优化目标

让用户快速识别：

- 高销售高利润率的优质团购
- 高销售低利润率的薄利团购
- 低销售高利润率的小众高效团购
- 退款偏高、利润承压的团购

### 6.4 具体方案

在现有列表页新增 `列表视图 / 盈利矩阵视图` 切换。

#### 方案 A：保留现有列表视图

保留当前能力：

- 搜索
- 排序
- 分页
- 点击查看详情

当前列表已经是很好的明细入口，不需要推翻，只需要补充更适合比较的视图。

#### 方案 B：新增盈利矩阵视图

建议使用气泡图：

- X 轴：销售额 `totalRevenue`
- Y 轴：利润率 `totalProfitMargin`
- 气泡大小：订单量 `totalOrderCount`
- 气泡颜色：
  - 利润为正：绿色
  - 利润接近 0：橙色
  - 利润为负：红色

图表交互：

- hover 显示：
  - 团购名称
  - 供货商
  - 销售额
  - 利润
  - 利润率
  - 订单量
  - 退款金额
- 点击气泡：
  - 打开当前已有的详情弹窗

#### 方案 C：增加快速筛选标签

基于当前已有字段，在图表顶部增加快速筛选：

- 仅看高退款团购
- 仅看低利润率团购
- 仅看亏损团购
- 仅看高销售团购

这些筛选都只基于当前列表数据前端计算，不要求新增接口字段。

### 6.5 为什么这样做

列表解决“找一个团购”的问题；
矩阵解决“比较一批团购”的问题。

二者用途不同，不应相互替代。

## 7. 团购详情盈利分析层

### 7.1 适用页面

- `frontend/src/pages/Analysis/components/GroupBuyOverview/components/MergedGroupBuyDetailModal.tsx`
- `frontend/src/pages/Analysis/components/GroupBuyOverview/components/GroupBuyHistoryAnalysis.tsx`

### 7.2 当前问题

当前详情弹窗已经有很多数据，但主线仍偏“指标堆叠”。

用户能够看到数字，但不容易形成如下判断：

- 当前盈利质量高不高
- 退款是不是主要问题
- 利润波动主要发生在哪几期
- 当前团购是“稳定赚钱”还是“偶发性赚钱”

### 7.3 优化目标

把详情页从“指标罗列”改为“盈利判断流程”：

1. 先看整体赚不赚钱
2. 再看退款是不是主要拖累
3. 再看历次发起是否稳定
4. 最后看客户与区域解释因素

### 7.4 具体方案

#### 方案 A：顶部核心区改为“盈利能力概览”

保留现有字段，但调整视觉主次。

主指标：

- 总销售额
- 总利润
- 利润率
- 退款金额

辅助指标：

- 总订单量
- 发起次数
- 平均每次销售额
- 平均每次利润

交互优化：

- 增加“金额模式 / 效率模式”切换
- 金额模式强调：
  - 总销售额
  - 总利润
  - 退款金额
- 效率模式强调：
  - 利润率
  - 平均每次利润
  - 平均每次订单量
  - 平均客单价

#### 方案 B：新增盈利拆解视图

使用当前已有字段，构建一个“解释型可视化区”。

建议表现形式：

- 四段拆解卡
  - 销售额
  - 退款金额
  - 利润
  - 利润率

或者使用简化瀑布表达：

- 净销售额
- 退款影响
- 利润结果

注意：

由于当前没有“原始销售额总额”字段，因此不做严格意义上的完整瀑布图，只做“盈利关系说明图”。

#### 方案 C：团购历史改为“图表优先，表格次级”

当前已有 `groupBuyLaunchHistory`，字段足够支持更好的图表表达。

建议新增两个可切换视图：

1. 历次发起组合图

- X 轴：发起时间
- 柱：
  - 销售额
  - 退款金额
- 线：
  - 利润
  - 或利润率

2. 历次发起散点图

- X 轴：发起时间
- Y 轴：利润率
- 点大小：订单量
- 点颜色：利润正负

用户可以快速看出：

- 哪些期销售额大，但利润率低
- 哪些期退款异常高
- 利润表现是否稳定

#### 方案 D：历史表格保留，但降级为明细层

保留当前 `GroupBuyHistoryAnalysis` 表格能力：

- 排序
- 展开全部
- 跳转团购详情

但调整展示顺序：

- 先图表
- 后表格

### 7.5 为什么这样做

当前详情数据最大的价值，不是继续做更多卡片，而是让用户看出“盈利结构和期次波动”。

这一步是本次方案里最关键的交互优化点。

## 8. 总览趋势效率层

### 8.1 适用页面

- `frontend/src/pages/Analysis/components/Overview/index.tsx`
- `frontend/src/pages/Analysis/components/Overview/components/UnifiedTrendChart.tsx`

### 8.2 当前问题

当前总览趋势能看出：

- 团购数趋势
- 订单趋势
- 销售额趋势
- 利润趋势

但不能直接回答：

- 盈利效率是在提升还是下降

### 8.3 优化目标

在不增加任何后端字段的情况下，补充“利润率趋势”视角。

### 8.4 具体方案

基于前端已有 `priceTrend` 和 `profitTrend` 计算：

- `profitMarginTrend = profitTrend / priceTrend`

展示方式：

- 在总览趋势区增加 `金额趋势 / 效率趋势` 切换

金额趋势：

- 团购数
- 订单数
- 销售额
- 利润

效率趋势：

- 利润率趋势

注意点：

- 当某一时间点销售额为 `0` 时，利润率按 `0` 展示
- 保持与现有“按日 / 按月 / 累计”逻辑一致

### 8.5 为什么这样做

销售额和利润告诉用户“结果大小”；
利润率告诉用户“赚钱效率”。

对于盈利能力判断，效率视角不能缺。

## 11. 派生计算口径

以下派生值均在前端基于现有字段计算，不新增接口字段。

### 11.1 规格定价能力

- 单份毛利：

`unitProfit = price - costPrice`

- 单份毛利率：

`unitProfitMargin = price > 0 ? (price - costPrice) / price : 0`

### 11.2 历史利润率

针对 `groupBuyLaunchHistory` 中的每一期：

`historyProfitMargin = revenue > 0 ? profit / revenue : 0`

### 11.3 总览利润率趋势

针对每个时间点：

`trendProfitMargin = price > 0 ? profit / price : 0`

### 11.4 退款压力标识

可在前端加轻量标签，不改变数据含义：

- 当 `totalRefundAmount > 0` 时显示“有退款”
- 当 `totalRefundAmount / totalRevenue` 较高时显示“退款偏高”

注意：

- 此类标签只做视觉辅助
- 具体阈值应在前端统一定义，避免页面间不一致

## 12. 视觉与交互规范

### 12.1 颜色规范

- 正利润：绿色
- 低利润或临界：橙色
- 负利润：红色
- 销售额主色：蓝色
- 退款主色：橙红色

### 12.2 交互规范

- 图表 hover 必须给出精确数值
- 图表点击若能进入详情，则必须保持与现有详情弹窗行为一致
- 所有排序规则默认按“最有业务价值”字段降序：
  - 规格默认按单份毛利降序
  - 团购矩阵默认展示全部，不改变原搜索条件
  - 历史默认按发起时间从新到旧

### 12.3 响应式规范

- 桌面端优先使用图表 + 表格组合
- 移动端图表优先使用可横向滚动或卡片化结构
- 不能让一个复杂图表在移动端挤压到不可读

## 13. 分阶段实施建议

### 第一阶段：低风险高收益

目标：最快让盈利能力更容易读懂。

实施项：

- 团购详情增加“规格定价能力表”
- 团购分析详情增加“历史盈利图表”
- 总览增加“利润率趋势”

收益：

- 改动小
- 不涉及后端
- 用户感知明显

### 第二阶段：比较能力增强

目标：让用户能快速发现“哪类团购值得关注”。

实施项：

- 团购概况页新增“盈利矩阵视图”
- 增加图表与列表视图切换
- 增加高退款、低利润率、亏损团购快速筛选

收益：

- 更适合批量经营分析
- 更接近管理驾驶舱场景

### 第三阶段：细节打磨

目标：统一视觉语言与交互一致性。

实施项：

- 统一利润颜色体系
- 统一 tooltip 文案
- 统一金额模式/效率模式切换
- 优化移动端布局

## 14. 风险与边界说明

### 14.1 当前方案不能表达的内容

由于不新增字段，当前方案不能严格表达：

- 规格级真实经营贡献利润
- 团购固定成本后的净利润
- 配送、包装、损耗、补贴等经营成本
- 区域维度的真实利润贡献

因此本次“盈利能力”本质上展示的是：

- 基于当前订单、售价、成本价、退款口径计算的贡献利润能力

### 14.2 需要避免的误导

页面文案不要暗示：

- 这是最终净利润
- 规格图能代表规格级真实经营贡献

更准确的表达应为：

- 利润
- 利润率
- 定价能力
- 历次发起盈利表现

### 14.3 数据一致性要求

所有展示必须遵循 `backend/README.statistics.md` 当前定义的统计口径，不允许前端另起一套解释规则。

## 15. 验收标准

实施完成后，应满足以下验收点：

### 15.1 规格层

- 用户可直接看出每个规格的单份毛利和毛利率
- 不需要人工心算

### 15.2 团购概况层

- 用户可通过图表快速识别高销售低利润率、亏损、高退款团购
- 点击图表与点击列表都能进入同一详情链路

### 15.3 团购详情层

- 用户可在 10 秒内判断某个团购是否值得继续做
- 用户可看出该团购表现是否稳定
- 用户可看出退款是否为主要问题

### 15.4 总览层

- 用户不仅能看到销售额和利润变化
- 还能看到利润率变化

### 15.5 一致性

- 所有金额、利润、退款、订单量均与当前接口口径一致
- 不出现前后端口径不一致的展示

## 16. 最终结论

在“只使用当前已有字段”的前提下，本项目完全可以把“团购盈利能力”从当前的静态数字展示，升级为一套更适合经营判断的可视化方案。

最值得优先落地的方向不是继续增加指标，而是：

- 把规格区升级为“定价能力展示”
- 把团购概况升级为“比较型视图”
- 把团购详情升级为“盈利结构 + 历史波动”视图
- 把总览趋势补上“利润率”效率视角

这套方案不依赖新增后端字段，改动边界清晰，业务收益明确，适合作为当前阶段的最短路径实施方案。
