# 注释详细程度保留规范

## 何时必须用

满足以下任一场景时，必须先参考本 Skill 再写或再改注释：

1. 会新增、修订、删除注释或 JSDoc。
2. 会新增或修改 `type`、`interface`、DTO、共享契约、Store state、Store action、工具函数、类方法等需要维护者先理解语义的结构。
3. 修改已有注释较多、较详细的文件，尤其是请求封装、并发、日志、数据库、统计、跨端契约等高约束逻辑。
4. 需要统一注释风格，但不确定哪些信息是噪音、哪些是有效上下文。

## 目标

让“注释详细程度”成为开发默认约束，而不是事后补票动作：保留对维护者真正有帮助的上下文，避免把“详细且有用”的注释压缩成“更短但信息更少”的版本，也避免新增注释时写得过浅。

## 核心原则

### 原则 1：注释服务于理解成本，不服务于篇幅整齐

- 判断标准不是“这段代码长不长”，而是“新人第一次看到时，能不能直接知道职责、输入输出、关键约束和为什么这样设计”。
- 可以精简纯粹复述字面行为、没有信息增量的注释，但不能为了统一篇幅把有效上下文一起删掉。

### 原则 2：先写语义与边界，再写表面行为

- 类型、字段、state、action、函数、方法的注释，优先解释业务语义、状态联动、跨端约束、副作用影响。
- “为什么这样设计”“不这样做会出什么问题”“这个字段在流程里扮演什么角色”优先级高于“这里做了赋值/调用/过滤”。

### 原则 3：注释粒度服从抽象度与维护风险

- 不只是长函数才需要细注释；承载多段语义的类型、聚合上下文、Store action、返回结果同样要写清楚。
- 简单 getter / mapper / 透传逻辑可以轻量；一旦涉及权限、异步编排、缓存、日志、数据库、统计、副作用聚合等非直观逻辑，就要提高注释密度。

### 原则 4：注释与实现同步交付

- 只要新增或修改类型定义、函数、方法、复杂 state/action，就默认评估并补齐注释，不把它当成后续治理任务。
- 写代码时已经识别出的协议约束、边界条件、跨端依赖、时序要求，要当下写进注释。

## 最低覆盖要求

- 新增或修改 `type` / `interface` / DTO / 配置对象时，字段若承载业务语义、来源差异、统计口径、跨端约束，默认补同行注释。
- 新增或修改 Store 时，核心 state 要说明“这份状态代表什么”，核心 action 要说明“它负责哪段业务流程 / 会同步哪些状态”。
- 新增或修改函数、方法时：
  - 单纯 getter、格式转换、常规透传：可用短 JSDoc 或保持轻量。
  - 涉及异步编排、缓存、日志、数据库、统计、副作用聚合等非直观逻辑：补 JSDoc，必要时再写分段注释说明关键步骤。

## 执行检查单

1. 动代码前先看当前文件已有注释，区分“噪音”与“有效上下文”。
2. 如果原注释已经解释了协议、边界、原因、风险，默认保留，只做修订、归位和术语统一。
3. 新增或修改类型、函数、方法、复杂 state/action 时，先回答“新人第一次看到时，需要知道哪些职责、输入输出、关键约束、状态联动”。
4. 调整注释后，自查是否丢失了原文件中的关键信息点，以及是否仍存在需要靠追调用才能看懂的声明。

## 反模式

- 不要把原本较详细的业务/协议注释删成一句表面描述。
- 不要为了统一风格，把不同风险级别的代码写成同一种注释密度。
- 不要把“保留详细程度”误解为保留过时信息；过时内容要修正，不是原样复制。
- 不要新增大量没有信息增量的注释来凑“详细”。
