# 注释业务语境与详细程度保留规范

## 何时必须用

满足以下任一场景时，必须先参考本 Skill 再写或再改注释：

1. 新增、修订、删除注释、JSDoc 或页面/组件文件头说明。
2. 新增或修改 `type`、`interface`、DTO、共享契约、Store state/action、Hook、工具函数、类方法。
3. 修改请求封装、并发、上传、审计日志、数据库、统计、前后端契约、页面复杂副作用等高约束逻辑。
4. 发现旧注释只解释技术动作，却没有说明业务来源、去向、触发场景、接口口径或维护边界。

## 目标

让注释能帮助接手开发者快速看出业务流程、输入输出、影响范围和边界。保留有效上下文，删改过时或无信息增量的噪音，不把复杂说明压缩成只剩技术动作的空话。

## 核心原则

### 原则 1：注释服务于理解成本，不服务于篇幅整齐

- 判断标准不是“代码长不长”，而是“接手开发者第一次看到时，能不能直接知道职责、输入输出、关键约束和为什么这样设计”。
- 可以精简纯粹复述字面行为、没有信息增量的注释，但不能为了统一篇幅把有效上下文一起删掉。
- 只要某个声明需要追多个调用点、上下游接口、状态联动或页面交互链路才能看懂，就说明注释仍然不够。

### 原则 2：先写业务语义与边界，再写表面行为

- 类型、字段、state、action、函数、Hook、方法的注释，优先解释业务语义、状态联动、接口口径、副作用影响、跨页面或跨层依赖。
- “为什么这样设计”“不这样做会出什么问题”“这个字段在流程里扮演什么角色”优先级高于“这里做了赋值/调用/过滤”。
- 写业务语义时不要脱离上下游：至少交代当前逻辑承接哪个页面/弹窗/接口/Store 状态，完成后影响哪个列表、详情、表单、日志或后端持久化口径。

### 原则 3：避免“技术孤岛型注释”

- 不要只写“获取详情”“执行搜索”“构建请求数据”“更新状态”“转换明细”“设置 loading”等技术动作；这些词可以出现，但必须补足业务对象、触发点和影响范围。
- 如果注释里的主语只能是“代码/函数/组件”，通常还不够；应尽量改成业务对象或流程主语。
- 注释不要堆实现细节替代业务解释。除非算法、并发、缓存、请求时序本身是风险点，否则不要逐行翻译代码。
- 注释中的业务名词要沿用页面、类型、接口和用户可见文案里的真实叫法，不确定时先追代码或询问用户。

### 原则 4：注释粒度服从抽象度与维护风险

- 不只是长函数才需要细注释；抽象名字但承载多段语义的类型、请求响应模型、Store action、页面派生状态、Hook 返回值同样要写清楚。
- 简单 getter、纯格式转换、明显的 JSX 结构可以保持轻量；一旦涉及权限、路由、异步编排、缓存、上传、审计日志、数据库、统计、状态持久化等非直观逻辑，就要提高注释密度。

### 原则 5：注释与实现同步交付

- 只要新增或修改类型定义、函数、Hook、方法、复杂 state/action，就默认评估并补齐注释。
- 写代码时已经识别出的接口约束、边界条件、状态时序、组件联动、错误处理策略，要当下写进注释。
- 已经打开并理解的旧文件，如果发现注释缺失、过时或只复述表面动作，可以在该文件内局部修订；不要扩大成全仓注释治理。

## 注释排版决策

1. 如果注释包含两句及以上内容，或承担背景、边界、状态联动、风险说明，保留独立 JSDoc / 分段注释。
2. 如果注释只有一句话，且声明本体能保持单行，优先使用同行尾注释。
3. 如果注释只有一句话，但声明签名较长、主体是函数 / Hook / 组件 / Store action，使用单行 `//`，不要默认写成三行 JSDoc。
4. 页面或组件中的业务事件入口方法，即使内部逻辑清晰，也优先保留一条简洁头部 JSDoc，说明它负责的业务流程、输入来源或状态联动。

## 最低覆盖要求

- 页面/组件文件头：说明职责和边界；职责、数据来源或交互边界变化时同步修订。
- 类型/API/DTO/Store/Hook/页面复杂逻辑：说明字段或 action 在业务流程里的含义、输入输出、接口口径和状态影响。
- 事件入口方法：说明承接哪段用户操作或弹窗流程；内部不要逐行翻译代码。
- 已读旧文件：发现当前文件内注释缺失、过时或只复述动作时，可以局部修订；不要扩展成全仓注释治理。

## 执行检查单

1. 动代码前先看当前文件已有注释，区分“噪音”“过时内容”和“有效上下文”。
2. 有效上下文默认保留，只做修订、归位和术语统一。
3. 新增或修改声明时，先回答“接手开发者第一次看到时，需要知道哪些业务职责、输入输出、关键约束和状态联动”。
4. 调整注释后，自查是否仍存在必须追调用才能看懂的业务语义。

## 反模式

- 不要把原本较详细的业务/协议注释删成“获取数据”“处理保存”“更新列表”“执行上传”之类的表面描述。
- 不要写脱离业务上下游的技术说明，例如只解释 Promise、state、filter、map、loading、request。
- 不要把“保留详细程度”误解为保留过时信息；过时内容要修正，不是原样复制。
- 不要把“允许顺手修订已读旧文件注释”扩大成跨模块批量改注释，或在注释修订中夹带无关业务代码变更。
- 不要新增大量没有信息增量的注释来凑“详细”。
