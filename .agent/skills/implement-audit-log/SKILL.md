# 审计日志实现规范

## 何时使用

以下场景必须使用本 Skill：

- 新增、修改、删除业务实体
- 关键状态流转，例如付款、完成、退款、部分退款
- 批量导入、批量创建、批量删除
- 导出、打印、敏感详情查看、关键统计查看
- 其他需要“事后追责”的操作

普通技术排错日志不属于本 Skill 的范围。`console`、`Logger`、接口报错输出不能替代审计日志。

## 先分清两类日志

### 1. 审计日志

用于回答“谁在什么时间，对什么对象，做了什么操作，结果如何”。

要求：

- 可持久化
- 可检索
- 可追责
- 字段结构稳定

### 2. 技术运行日志

用于排查异常、性能和系统问题。

要求：

- 可帮助排查问题
- 不要求承担业务审计职责

不要把这两类日志混在一起设计。

## fresh-shop 中的实现原则

### 原则 1：以后端为准

审计日志优先在后端落库或进入统一后端记录链路。前端埋点、控制台输出、第三方统计只能作为补充，不能作为唯一审计来源。

### 原则 2：在业务动作附近记录

日志应写在真正完成业务语义的地方，通常是 service 层，而不是仅写在 controller 或页面点击事件里。

### 原则 3：只记录对追责有价值的信息

日志要能说明发生了什么，但不要把整份大对象、二进制内容或无关噪声全部写进去。

### 原则 4：成功和失败要区分

- 成功操作：记录最终生效的业务结果
- 失败操作：只在业务上需要追踪失败尝试时记录，例如退款失败、批量导入部分失败、非法状态流转

## 查询类操作的边界

“查”不等于所有列表请求都必须写审计日志。按价值分层：

- 需要记录：详情查看、导出、打印、关键筛选查询、关键统计查看、敏感数据查询
- 通常不记录：页面初始化的普通列表加载、自动轮询、前端重复刷新、无业务意义的被动请求

如果某个查询是“用户明确发起且事后需要追责”的，就应进入审计范围。

## 推荐字段

设计审计日志时，至少覆盖以下字段。详细说明见 `references/audit-log-fields.md`。

- `module`：业务模块，例如 `order`、`groupBuy`
- `action`：操作动作，例如 `create`、`update`、`refund`、`export`
- `targetType`：目标对象类型
- `targetId`：目标对象主键；批量操作可为空，但要有批次标识
- `operatorId`
- `operatorName`
- `result`：`SUCCESS` / `FAILED`
- `summary`：一句话说明本次操作
- `before`：变更前的关键字段快照
- `after`：变更后的关键字段快照
- `meta`：查询条件、批次统计、来源页面等补充上下文
- `createdAt`

## 写入时机

### 新增

在业务实体创建成功后记录，重点保留：

- 创建人
- 创建对象
- 关键业务字段摘要

### 修改

在更新成功后记录，重点保留：

- 修改前关键字段
- 修改后关键字段
- 为什么变更或从哪个入口触发

### 删除

在删除成功后记录，重点保留：

- 被删除对象的身份信息
- 删除前关键摘要

### 状态流转

状态流转优先于普通 update 单独建 action，例如：

- `pay`
- `complete`
- `refund`
- `partial-refund`

不要把关键状态迁移简单塞进通用 `update`，否则后续检索成本很高。

### 批量操作

批量操作至少记录：

- 批次维度摘要
- 成功数 / 失败数
- 失败原因汇总

必要时再补单条子记录，不要默认把每条原始 payload 全量落库。

## 前后端职责分工

### 前端

- 负责提供必要的操作上下文，例如当前页面、筛选条件、触发入口
- 不在成功结果未确认前写“成功”审计
- 不把前端埋点当成审计主链路

### 后端

- 负责生成最终可信的审计事件
- 负责补足操作者、目标对象、结果、前后快照
- 负责控制敏感字段脱敏与存储边界

## 敏感信息与体积控制

以下内容默认不进入审计日志，除非业务明确要求且已脱敏：

- token、密码、密钥、认证头
- 超大富文本、图片二进制、文件原文
- 无法检索且没有追责价值的整对象冗余快照

快照只保留关键业务字段，不要把 Prisma 查询结果整体塞进日志。

## 落地建议

若当前仓库还没有统一审计模块，新增能力时优先考虑：

- 在后端增加独立的审计日志模块或服务
- 由业务 service 调用统一的 audit service 写入
- 共享字段定义放在 `backend/types/` 或审计模块内部统一维护
- 若只是局部引入，不要把日志结构散落在多个业务 service 中各写一套

不要在每个业务模块里各自手写一套随意的日志结构。

## 自查清单

完成后至少确认：

- 是否真的记录到了“谁、何时、对谁、做了什么、结果如何”
- 关键状态流转是否被单独建模
- 查询类日志是否只覆盖有追责价值的动作
- 快照是否只保留关键字段
- 是否避免了前端先写成功、后端实际失败的错位
