# 项目开发工作流

## 何时用

- 新功能、跨前后端联动、范围不清的任务
- 需要先做验收契约、TODO 拆解、验证计划
- 会影响多个页面、Store、接口、共享类型或数据库实体的任务

## 最小输出

1. 任务目标、输入输出、边界情况
2. 受影响文件 / 模块 / 页面 / 接口 / 数据实体
3. TODO 拆解与可回滚点
4. 验证方案
5. 风险点与是否需要转入其他领域 Skill

## 阶段 0：理解需求与收集上下文

先确认以下问题，再开始改代码：

- 业务目标是什么
- 改动边界是什么
- 涉及哪些页面、Store、接口、数据库实体或共享类型
- 是否存在统计口径、状态流转、权限、审计日志等隐含约束

执行要求：

- 至少查找 3 个项目内相似实现，优先复用已有模式
- 若需求关键约束不清晰，先向用户澄清
- 若目标和边界已经清晰，则直接执行，不要为了流程而阻塞任务

## 阶段 1：形成执行方案

在动手前先收敛以下内容：

- 修改哪些文件
- 每个文件承担什么改动
- 是否涉及共享类型、接口契约或统计口径
- 需要怎样验证

对中大型改动，应先把计划同步给用户；对边界清晰的小改动，可直接落地并在执行过程中持续同步。

## 阶段 2：代码执行

执行时遵循以下原则：

- 小步迭代，避免一口气扩散到无关模块
- 优先沿用现有目录结构、命名方式、错误处理风格和注释风格
- 前端业务逻辑优先进入 Store，不把请求编排塞进页面组件
- 后端 controller 保持轻量，业务规则集中在 service
- 涉及共享数据结构时，优先更新 `backend/types/`
- 涉及统计分析时，以 `backend/README.statistics.md` 为准
- 规划完成后转入对应领域 Skill：
  - 前端走 `create-ui-component` / `create-zustand-store`
  - 后端走 `create-backend-module` / `prisma-workflow` / `implement-audit-log`
  - 共享契约走 `type-contract-guidelines`
  - 注释与维护信息走 `comment-detail-preservation`

## 阶段 3：验证与交付

验证默认遵循“按改动范围最小充分验证”原则，优先级为：

- 代码逻辑自检 / 受影响链路闭环检查
- 定向 lint / typecheck
- 测试

至少确认：

- 代码逻辑是否自洽
- 受影响的调用链是否闭环
- 注释是否已同步更新
- 是否真的需要运行命令；`install / dev / build / preview / start` 不是默认步骤

交付时至少说明：

- 改了什么
- 为什么这样改
- 做了哪些验证
- 还有哪些剩余风险或前置假设

## 反模式

- 不给补丁式兼容方案
- 不做超出需求的附加设计
- 不跳过验收契约直接开改
- 不在没有相似实现时自造新模式
- 不静默执行破坏性操作
- 不默认执行 `dev`、`build`、`install` 类命令
