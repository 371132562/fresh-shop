# Ant Design 文档使用规范

## 何时用

- 不确定某个 Ant Design 组件的 props、事件、受控方式或组合模式
- 需要按照官方组件能力设计交互，而不是靠猜测 prop 名
- 需要确认弹窗、表单、列表、上传、通知等组件的推荐用法

## 执行检查单

1. 优先以 Ant Design 官方文档为准，不凭记忆臆造 API。
2. 结论要结合当前项目实际版本与既有模式：项目使用 Ant Design 6，页面多数以 `Form`、`List`、`Modal`、`Card`、`notification` 为基础。
3. 如果官方文档示例与当前项目版本或代码风格存在差异，优先给出当前项目可落地的等价写法。
4. 不要为了追求“官方推荐”而打破当前项目已稳定的交互模式，除非现有模式本身有明显问题。
5. 组件选型要回答清楚：为什么选它，而不是另一个视觉相近的组件。

## 反模式

- 不要凭印象发明不存在的 props 或事件。
- 不要把官方示例原样硬塞进当前项目而忽略现有请求层、Store 层和样式结构。
- 不要把 Ant Design 文档当成业务规范来源；业务边界仍以项目代码与用户需求为准。
